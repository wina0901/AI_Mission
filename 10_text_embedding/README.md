# 20 Newsgroups 뉴스 문서 분류 프로젝트

## 프로젝트 개요

이 프로젝트는 20 Newsgroups 데이터셋을 활용해 뉴스 문서를 20개의 카테고리로 분류하는 텍스트 분류 프로젝트입니다.  
단순히 문장을 벡터화하고 분류 모델을 적용하는 데서 끝나는 것이 아니라, 텍스트 전처리, 단어 사전 구축, 임베딩 방식 비교, BiGRU 기반 딥러닝 모델 설계, 하이퍼파라미터 튜닝, 테스트 평가, 클래스별 오류 분석까지 전체 자연어처리 분류 파이프라인을 직접 구현하는 데 집중했습니다.

특히 이번 프로젝트에서는 Word2Vec, FastText, GloVe를 각각 임베딩으로 활용해 같은 BiGRU 분류기 위에서 성능을 비교했고, 가장 좋은 임베딩 조합을 선택한 뒤 추가 튜닝을 진행했습니다.  
또한 전체 정확도만 보는 것이 아니라, 특정 클래스가 왜 유독 잘 안 맞는지까지 분석하며 모델의 한계와 데이터 특성을 함께 해석했습니다.

---

## 이 프로젝트를 왜 했는가

텍스트 분류는 자연어처리에서 가장 기본적이면서도 중요한 문제 중 하나입니다.  
뉴스 카테고리 분류는 이메일 필터링, 문서 자동 분류, 고객 문의 분류, 게시글 분류 같은 실제 서비스 문제와 구조가 매우 유사하기 때문에, 실무형 NLP 문제를 경험하기에 적절한 주제라고 생각했습니다.

이번 프로젝트는 다음과 같은 이유로 진행했습니다.

- 자연어처리에서 텍스트 전처리부터 딥러닝 분류 모델까지 전체 흐름을 직접 경험해보고 싶었습니다.
- 단순한 Bag-of-Words 기반 분류가 아니라, 단어 임베딩과 순차 모델을 활용한 딥러닝 기반 텍스트 분류를 구현해보고 싶었습니다.
- Word2Vec, FastText, GloVe처럼 서로 다른 임베딩 방식이 실제 분류 성능에 어떤 차이를 만드는지 비교해보고 싶었습니다.
- 단순 정확도 비교에 그치지 않고, 어떤 클래스가 왜 성능이 낮은지 오류 분석까지 수행해보고 싶었습니다.
- 전처리, 모델 비교, 튜닝, 해석까지 포함한 포트폴리오형 NLP 프로젝트를 완성하고 싶었습니다.

즉, 이번 프로젝트는 단순한 뉴스 분류 실습이 아니라, 텍스트 데이터를 실제로 어떻게 정제하고 표현하며, 어떤 임베딩과 모델 조합이 더 적절한지 검증하는 과정을 직접 경험하기 위해 진행했습니다.

---

## 문제 정의

- 주제: 뉴스 문서 카테고리 분류
- 데이터셋: 20 Newsgroups
- 문제 유형: 멀티클래스 텍스트 분류
- 입력: 뉴스 문서 텍스트
- 출력: 20개 카테고리 중 하나의 클래스 예측
- 주요 평가 지표:
  - Accuracy
  - Precision
  - Recall
  - F1-score

이번 프로젝트의 핵심은 문서를 단순히 숫자로 바꾸는 것이 아니라, 문맥 정보를 반영할 수 있는 임베딩과 시퀀스 모델을 활용해 뉴스의 주제를 정확히 분류하는 것이었습니다.

---

## 데이터 설명

사용한 데이터는 sklearn에서 제공하는 20 Newsgroups 데이터셋입니다.

데이터 특성은 다음과 같습니다.

- 총 20개 뉴스 카테고리로 구성
- 각 문서는 하나의 카테고리 라벨을 가짐
- 헤더, 푸터, 인용문을 제거한 텍스트 사용
- 비교적 긴 문서와 짧은 문서가 혼재
- 클래스 분포는 대체로 균형적이지만, 주제 간 의미가 겹치는 클래스가 존재

이번 프로젝트에서는 뉴스 문서 특유의 불필요한 메타데이터를 줄이기 위해 headers, footers, quotes를 제거한 버전을 사용했습니다.

---

## 프로젝트 진행 과정

### 1. 데이터 로드 및 기초 탐색

먼저 20 Newsgroups 데이터를 불러오고, 전체 문서 수와 클래스 분포를 확인했습니다.  
이후 클래스별 문서 개수를 시각화해 특정 클래스에 심한 불균형이 없는지 점검했습니다.

이 단계에서 중요한 점은 모델을 바로 학습시키는 것이 아니라, 데이터가 어떤 구조를 가지고 있고 각 카테고리가 어느 정도의 샘플 수를 가지는지 먼저 확인하는 것이었습니다.

---

### 2. 텍스트 전처리

텍스트 데이터는 원문 그대로 사용하지 않고, 뉴스 문서 분류에 불필요한 요소를 줄이기 위한 전처리를 수행했습니다.

적용한 주요 전처리는 다음과 같습니다.

- 소문자 변환
- 헤더 형식 제거
- 이메일 주소, 숫자, 특수문자 정리
- 토큰화
- 불용어 제거 여부를 조절 가능한 형태로 구성

전처리 함수는 단순 정리용이 아니라, 문서 본문의 핵심 단어를 최대한 남기면서 분류에 방해가 되는 잡음을 줄이기 위한 방식으로 설계했습니다.

즉, 이번 프로젝트는 텍스트 전처리를 기계적으로 수행한 것이 아니라, 뉴스 문서라는 데이터 특성에 맞게 불필요한 정보를 줄이는 방향으로 설계한 프로젝트였습니다.

---

### 3. 데이터 분할 및 시퀀스 준비

학습 안정성과 성능 검증을 위해 데이터를 train, validation, test로 나누어 사용했습니다.

- train / test 분할
- train 내부에서 validation 추가 분할
- stratify 적용으로 클래스 비율 유지

이후 각 문서를 토큰 시퀀스로 변환하고, 학습 데이터 기준으로 단어 사전을 구축했습니다.

주요 설정은 다음과 같습니다.

- min_freq = 2
- max_vocab_size = 30000
- 특수 토큰 포함
  - <PAD>
  - <UNK>

또한 문장 길이 분포를 먼저 분석한 뒤, 95퍼센타일 기준을 참고해 max_len을 설정하고 패딩을 적용했습니다.  
이 과정을 통해 지나치게 긴 문장으로 인한 비효율은 줄이면서도 대부분의 문서 정보를 유지하도록 설계했습니다.

---

### 4. 임베딩 구성

이번 프로젝트의 핵심 실험 중 하나는 서로 다른 단어 임베딩을 비교하는 것이었습니다.

비교한 임베딩은 다음과 같습니다.

- Word2Vec
- FastText
- GloVe

각 임베딩은 다음과 같은 방식으로 준비했습니다.

#### Word2Vec
학습용 문서 토큰을 사용해 직접 Word2Vec 임베딩을 학습했습니다.

- embedding dimension: 200
- sg=1
- epochs=20

#### FastText
학습용 문서 토큰으로 FastText 임베딩을 학습했습니다.  
FastText는 subword 정보를 반영할 수 있기 때문에, 희귀 단어나 형태 변화에 더 강할 가능성을 확인해보고자 했습니다.

#### GloVe
Stanford GloVe 사전학습 벡터를 불러와 vocabulary에 맞는 임베딩 행렬을 구성했습니다.

이후 각 임베딩을 동일한 BiGRU 분류기에 넣어 성능을 비교했습니다.  
즉, 모델 구조는 동일하게 유지하고 입력 표현 방식만 바꿔, 임베딩의 효과를 공정하게 비교하도록 설계했습니다.

---

### 5. BiGRU 기반 텍스트 분류 모델 설계

모델은 BiGRU 기반의 텍스트 분류기로 구성했습니다.

주요 구조는 다음과 같습니다.

- Embedding layer
- BiGRU
- Dropout
- Fully Connected layer
- Softmax classification output

양방향 GRU를 사용한 이유는 문장의 앞뒤 문맥을 함께 반영해 분류 성능을 높이기 위해서였습니다.  
단순 평균 임베딩보다 순차 구조를 갖는 RNN 계열 모델이 뉴스 문서처럼 길이가 있는 텍스트 분류에 더 적절하다고 판단했습니다.

---

### 6. 학습 및 평가 로직 구성

학습 과정에서는 다음 요소들을 직접 구현했습니다.

- train_one_epoch
- validation 평가 함수
- Accuracy / Precision / Recall / F1-score 계산
- EarlyStopping
- 실험별 history 저장
- 학습 곡선 시각화

이를 통해 단순히 epoch를 반복하는 것이 아니라, validation 성능이 더 이상 개선되지 않을 경우 조기 종료해 과적합을 줄이고, 실험별 학습 흐름을 비교할 수 있도록 구성했습니다.

---

## 기본 비교 실험 결과

세 가지 임베딩을 같은 BiGRU 분류기에 적용해 validation 성능을 비교했습니다.

그 결과, Word2Vec 기반 모델이 가장 높은 Validation F1-score를 기록해 최종 튜닝 대상으로 선정되었습니다.

이번 비교 실험의 의미는 단순히 어떤 임베딩이 더 좋았는지를 확인하는 데 그치지 않았습니다.  
같은 모델 구조에서도 입력 표현 방식에 따라 성능 차이가 발생할 수 있다는 점을 직접 검증했다는 점에서 의미가 있었습니다.

---

## 하이퍼파라미터 튜닝

기본 실험에서 가장 성능이 좋았던 Word2Vec 기반 모델에 대해 추가 하이퍼파라미터 튜닝을 진행했습니다.

주요 탐색 대상은 다음과 같습니다.

- dropout
- learning rate
- embedding freeze 여부
- hidden dimension 등 모델 설정 조합

실험 결과, 튜닝된 Word2Vec 기반 BiGRU 모델이 최종적으로 가장 좋은 성능을 보였습니다.

즉, 이번 프로젝트는 처음부터 하나의 모델만 고정해서 사용한 것이 아니라,  
임베딩 비교 → 최고 임베딩 선택 → 추가 튜닝이라는 단계적 방식으로 성능을 개선해나간 프로젝트였습니다.

---

## 최종 테스트 성능

테스트셋 기준 최종 선택 모델은 Word2Vec_Tuned 모델이었습니다.

최종 성능은 다음과 같습니다.

- Accuracy: 0.7379
- F1-score: 0.7372

또한 전체 테스트 샘플 3770개 중 2782개를 정확히 분류했습니다.

이 결과는 20개 클래스로 구성된 멀티클래스 뉴스 분류 문제에서, 전처리와 임베딩 비교, BiGRU 구조, 튜닝을 종합적으로 적용해 안정적인 수준의 성능을 확보했다는 의미가 있습니다.

---

## 클래스별 성능 분석

전체 성능이 나쁘지 않았더라도, 모든 클래스가 고르게 잘 분류된 것은 아니었습니다.  
그래서 이번 프로젝트에서는 클래스별 정확도를 따로 계산하고, 유독 성능이 낮은 클래스를 별도로 분석했습니다.

특히 낮은 성능을 보인 대표 클래스는 다음과 같았습니다.

- talk.religion.misc

이 클래스는 테스트 데이터 126개 중 26개만 정확히 분류되어 약 20.6%의 낮은 정확도를 보였습니다.

---

## 오류 분석

성능이 낮은 클래스에 대해서는 단순히 점수만 확인하지 않고, 왜 오분류가 많이 발생했는지를 추가로 분석했습니다.

talk.religion.misc 클래스의 주요 오분류 경향은 다음과 같았습니다.

- soc.religion.christian 으로 자주 혼동
- alt.atheism 으로 자주 혼동

즉, 전체 오답의 상당 부분이 종교 관련 주제가 겹치는 클래스들 사이에서 발생했습니다.

추가 분석을 통해 확인한 점은 다음과 같습니다.

- 해당 클래스는 다른 종교/신념 관련 클래스와 어휘가 많이 겹침
- 문서 길이 특성만으로는 구분이 쉽지 않음
- 상위 단어 역시 종교 관련 일반 표현이 많아 클래스 경계가 모호함

이 결과는 모델의 한계이기도 하지만, 동시에 데이터셋 자체에서 일부 클래스가 의미적으로 매우 가깝기 때문에 생기는 자연스러운 어려움이라는 점도 보여주었습니다.

즉, 이번 프로젝트는 단순히 성능 수치를 제시하는 데서 끝나지 않고,  
어떤 클래스에서 왜 틀리는지를 데이터 관점에서 해석했다는 점이 중요한 의미를 가집니다.

## 이 프로젝트를 통해 갖추게 된 능력

### 1. NLP 텍스트 분류 파이프라인을 end-to-end로 구현하는 능력

데이터 로드, 전처리, vocabulary 구축, 정수 인코딩, 패딩, 임베딩 행렬 생성, 모델 학습, 평가, 오류 분석까지 전체 자연어처리 분류 파이프라인을 직접 구현할 수 있게 되었습니다.

### 2. 단어 임베딩을 비교하고 활용하는 능력

Word2Vec, FastText, GloVe를 직접 구성하고 동일한 분류기 위에서 비교하면서, 단어 표현 방식이 실제 성능에 어떤 영향을 주는지 분석할 수 있었습니다.  
이를 통해 NLP에서 임베딩이 단순한 입력 벡터가 아니라 모델 성능에 큰 영향을 주는 핵심 요소라는 점을 실전적으로 이해할 수 있었습니다.

### 3. RNN 계열 텍스트 분류 모델 구현 능력

BiGRU 기반 분류 모델을 직접 설계하고 학습시키면서, 순차 정보를 활용하는 텍스트 분류 모델을 구현하는 역량을 쌓을 수 있었습니다.

### 4. 하이퍼파라미터 튜닝 및 성능 개선 능력

기본 모델 비교에 그치지 않고, 가장 성능이 좋은 임베딩 조합을 선택한 뒤 dropout, learning rate 등 주요 하이퍼파라미터를 조정해 성능을 개선했습니다.  
이를 통해 실험을 단계적으로 설계하고 성능을 개선하는 접근 방식을 경험할 수 있었습니다.

### 5. 클래스별 오류 분석 능력

전체 Accuracy만 확인하는 것이 아니라, 클래스별 정확도와 혼동 경향을 따로 분석했습니다.  
특히 특정 클래스의 오분류 패턴, 상위 단어, 문서 길이 특성을 함께 확인하면서, 모델 실패 원인을 데이터와 문제 구조 관점에서 해석하는 능력을 기를 수 있었습니다.

### 6. 모델 성능을 숫자와 해석으로 함께 설명하는 능력

이번 프로젝트에서는 단순히 최종 점수만 제시한 것이 아니라,  
왜 특정 임베딩이 더 좋았는지, 왜 특정 클래스는 어려웠는지까지 설명할 수 있도록 정리했습니다.  
이를 통해 결과를 단순 보고가 아니라 분석으로 연결하는 역량을 강화할 수 있었습니다.

## 이 프로젝트에서 중요하게 본 점

이번 프로젝트에서 가장 중요하게 생각한 점은 텍스트 분류 성능을 올리는 것 자체보다,  
어떤 표현 방식과 모델 구조가 더 적절한지 비교하고, 왜 특정 클래스에서 어려움이 생기는지를 해석하는 것이었습니다.

그래서 다음과 같은 원칙으로 프로젝트를 진행했습니다.

- 텍스트 전처리를 문제 특성에 맞게 설계하기
- 단어 사전과 시퀀스 길이를 데이터 분포를 보고 결정하기
- Word2Vec, FastText, GloVe를 같은 조건에서 공정하게 비교하기
- 같은 분류기 구조 위에서 임베딩 효과를 검증하기
- Validation 기준으로 최고 조합을 선택한 뒤 추가 튜닝하기
- 테스트셋 전체 성능뿐 아니라 클래스별 오류까지 분석하기

이러한 흐름은 향후 문서 분류, 고객 문의 분류, 게시글 자동 태깅, 뉴스 분류 같은 실무형 NLP 문제를 다룰 때 중요한 기반이 된다고 생각합니다.

## 개선 방향

이번 결과를 바탕으로 향후 다음과 같은 방향으로 성능을 더 높일 수 있다고 판단했습니다.

- Attention 메커니즘을 추가한 BiGRU 실험
- Transformer 계열 모델과의 성능 비교
- subword 또는 sentencepiece 기반 토크나이징 적용
- 클래스별 가중치 조정
- 데이터 증강 또는 back translation 기반 문장 확장
- 의미적으로 가까운 클래스에 대한 계층적 분류 실험

즉, 현재 모델은 전통적인 임베딩 + RNN 구조 기반으로 안정적인 성능을 보였지만, 더 강한 문맥 표현 모델을 적용하면 추가 성능 향상 가능성이 충분하다고 볼 수 있습니다.
