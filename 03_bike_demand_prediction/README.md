# 공유 자전거 수요 예측 프로젝트

## 프로젝트 개요

이 프로젝트는 시간, 계절, 날씨, 온도, 습도, 풍속과 같은 다양한 환경 정보를 바탕으로 공유 자전거 대여 수요를 예측하는 머신러닝 프로젝트입니다.  
단순히 모델을 한 번 학습시키는 데 그치지 않고, 데이터 탐색부터 전처리, 변수 해석, 모델 비교, 정규화 적용, 하이퍼파라미터 탐색까지 전체 예측 파이프라인을 직접 설계하고 검증하는 과정에 집중했습니다.

최종 목표는 RMSLE를 최소화하는 것이었으며, 여러 회귀 모델을 비교한 뒤 가장 적절한 모델을 선정해 테스트 데이터 예측까지 수행했습니다.

---

## 이 프로젝트를 왜 했는가

실무에서의 데이터 분석과 머신러닝은 단순히 모델 성능만 높이는 작업이 아니라, 데이터를 이해하고 문제에 맞는 방식으로 가공하며, 결과를 해석 가능한 형태로 정리하는 과정이 더 중요하다고 생각했습니다.

이번 프로젝트는 이러한 흐름을 직접 경험하기 위해 진행했습니다.

- 예측 문제에서 어떤 전처리가 실제 성능 향상에 도움이 되는지 확인하고 싶었습니다.
- 데이터의 분포와 변수 간 관계를 먼저 해석한 뒤 모델링에 반영하는 경험을 쌓고 싶었습니다.
- 단일 모델을 바로 선택하는 것이 아니라, 선형 회귀, 다항 회귀, 정규화 기법을 비교하며 모델 선택의 근거를 명확히 만들고 싶었습니다.
- 성능 수치만 보는 것이 아니라, 과적합 여부와 일반화 가능성까지 함께 판단하는 연습을 하고 싶었습니다.

즉, 이번 프로젝트는 단순한 예측 실습이 아니라, 데이터 기반으로 문제를 해석하고 더 나은 모델을 선택하는 사고 과정을 훈련하기 위해 수행한 프로젝트입니다.

---

## 데이터 및 문제 정의

- 주제: 공유 자전거 대여 수요 예측
- 목표: 대여 수량(count) 예측
- 평가지표: RMSLE (Root Mean Squared Logarithmic Error)

예측 대상은 count이며, 입력 변수로는 날짜, 계절, 휴일 여부, 근무일 여부, 날씨, 온도, 체감온도, 습도, 풍속 등이 활용되었습니다.

---

## 프로젝트 진행 과정

### 1. EDA를 통한 데이터 이해

먼저 train/test 데이터의 구조를 확인하고, 결측값과 중복값 여부를 점검했습니다.  
그 결과, 결측치와 중복 데이터는 존재하지 않았으며, 이후에는 변수별 분포와 관계를 중심으로 데이터를 분석했습니다.

주요 분석 내용은 다음과 같습니다.

- 시간대별 평균 대여량 분석
- 온도 및 체감온도 구간별 대여량 분석
- 습도별 대여량 분석
- 연도별, 월별 대여량 분석
- 풍속과 대여량의 관계 분석
- 날씨 상태별 대여량 분석
- 평일과 휴일 간 대여량 비교
- 전체 변수 간 상관관계 분석

이 과정을 통해 출퇴근 시간대에 이용량이 높고, 기온이 높을수록 대여량이 증가하며, 습도가 높을수록 대여량이 감소하는 경향을 확인했습니다. 또한 날씨가 좋지 않을수록 이용량이 줄어드는 패턴도 확인할 수 있었습니다.

---

### 2. 문제 특성에 맞는 전처리 수행

예측 성능을 높이기 위해 데이터 특성에 맞는 전처리를 적용했습니다.

수치형 변수 처리
- windspeed는 분포가 치우쳐 있어 로그 변환을 적용
- temp, atemp, humidity, windspeed는 표준화 수행
- 목표변수 count 역시 로그 변환하여 모델 안정성을 확보

범주형 변수 처리
- season, holiday, workingday, weather 변수에 대해 원-핫 인코딩 수행
- train/test 간 컬럼 불일치를 방지하기 위해 align 처리

날짜형 변수 처리
- datetime을 datetime 타입으로 변환
- year, month, day, hour 파생 변수 생성

추가 처리
- weather 값이 4인 경우 데이터 수가 매우 적어 모델 안정성을 위해 제거
- 모델 입력에서 casual, registered는 count와 구조적으로 직접 연결되므로 제외

이 과정은 단순한 전처리가 아니라, 데이터 분포와 변수 의미를 고려한 문제 맞춤형 피처 엔지니어링 과정이었습니다.

---

### 3. 모델 비교를 통한 최적 해 탐색

여러 회귀 모델을 직접 비교하면서 성능과 일반화 가능성을 함께 점검했습니다.

비교한 모델은 다음과 같습니다.

- 다중 선형 회귀
- 다항 회귀
- Ridge 기반 L2 정규화 모델
- Lasso 기반 L1 정규화 모델

주요 결과는 다음과 같았습니다.

- 다중 선형 회귀: 검증 RMSLE 0.9853
- 다항 회귀 2차: 검증 RMSLE 0.8229
- 다항 회귀 3차: 검증 RMSLE 0.7481
- 다항 회귀 3차 + L2 정규화(alpha=100): 검증 RMSLE 0.7367

단순 선형 모델보다 다항 회귀가 훨씬 나은 성능을 보였고, 3차 다항식에서는 성능은 좋아졌지만 과적합 가능성도 함께 나타났습니다. 이를 보완하기 위해 정규화를 적용했고, 그 결과 L2 정규화가 가장 안정적인 성능을 보였습니다.

---

### 4. Optuna를 활용한 하이퍼파라미터 최적화

모델 선택을 감각적으로 결정하지 않고, 하이퍼파라미터 탐색을 통해 객관적으로 검증했습니다.

- 탐색 방식: Optuna
- 탐색 대상: 다항식 차수, Ridge alpha
- 최적 결과:
  - degree = 3
  - alpha ≈ 99.53
  - best validation RMSLE = 0.736654

기존 실험에서 확인한 결과와 Optuna 탐색 결과가 거의 일치했기 때문에, 수작업 실험과 자동 탐색이 모두 같은 결론을 지지한다는 점도 확인할 수 있었습니다.

---

## 최종 모델

최종 선택 모델은 다음과 같습니다.

- Polynomial Regression
- degree = 3
- Ridge(L2) Regularization
- alpha = 100

이 모델을 선택한 이유는 다음과 같습니다.

- 비교 모델 중 가장 낮은 검증 RMSLE를 기록
- 비정규화 3차 다항 회귀보다 과적합이 완화됨
- 단순 선형 모델보다 훨씬 높은 설명력 확보
- 수작업 비교와 Optuna 최적화 결과가 일관되게 같은 방향을 가리킴

최종적으로는 train과 validation 데이터를 다시 합쳐 재학습한 후 test 데이터에 예측을 수행했습니다. 예측 결과에서 음수값은 발생하지 않았으며, 물리적으로 불가능한 음수 수요가 나올 가능성까지 고려해 후처리 로직도 반영했습니다.

## 이 프로젝트를 통해 갖추게 된 능력

### 1. 문제를 데이터 관점에서 해석하는 능력

단순히 모델을 적용하는 것이 아니라, 먼저 데이터 구조와 변수 의미를 파악하고 어떤 변수가 중요한지 해석하는 훈련을 했습니다.  
이를 통해 예측 문제를 숫자 맞추기 수준이 아니라, 데이터 기반 의사결정 문제로 바라보는 시각을 갖추게 되었습니다.

### 2. EDA 기반의 전처리 설계 능력

분포를 확인한 뒤 로그 변환과 표준화를 적용하고, 범주형 변수 인코딩과 날짜 파생 변수 생성까지 직접 수행했습니다.  
즉, 정형 데이터를 모델이 잘 학습할 수 있는 형태로 바꾸는 전처리 역량을 실제로 적용해볼 수 있었습니다.

### 3. 모델 성능 비교 및 일반화 판단 능력

여러 모델을 직접 비교하면서 단순한 훈련 성능이 아니라 검증 성능과 과적합 여부를 함께 확인했습니다.  
이를 통해 모델 선택 시 성능 수치만 보는 것이 아니라, 왜 그 모델이 더 적절한지 설명할 수 있는 역량을 기를 수 있었습니다.

### 4. 정규화와 하이퍼파라미터 최적화 활용 능력

다항 회귀에서 발생할 수 있는 과적합 문제를 L1, L2 정규화를 통해 비교했고, Optuna를 활용해 최적 파라미터를 탐색했습니다.  
이 과정에서 모델 개선이 단순 반복이 아니라, 구조적 문제를 파악하고 해결하는 방식이어야 한다는 점을 배웠습니다.

### 5. 머신러닝 프로젝트 전체 흐름을 끝까지 구현하는 능력

EDA → 전처리 → 분석 → 모델링 → 튜닝 → 최종 예측까지 전 과정을 스스로 진행했습니다.  
이 경험을 통해 단편적인 알고리즘 사용이 아니라, 하나의 예측 프로젝트를 처음부터 끝까지 완성하는 실무형 흐름을 익힐 수 있었습니다.

## 프로젝트에서 중요하게 본 점

이번 프로젝트에서 가장 중요하게 생각한 부분은 단순히 가장 좋은 점수를 얻는 것이 아니라, 왜 그런 결과가 나왔는지를 설명할 수 있는가였습니다.

그래서 다음과 같은 원칙으로 프로젝트를 진행했습니다.

- 데이터 해석 없이 바로 모델링하지 않기
- 전처리의 이유를 분명히 하기
- 여러 모델을 비교하고 근거 기반으로 선택하기
- 과적합 여부를 함께 확인하기
- 자동 탐색 결과와 직접 실험한 결과를 함께 검증하기

이러한 접근은 향후 데이터 분석가나 AI 엔지니어로서 실무 문제를 다룰 때도 중요한 기반이 된다고 생각합니다.
