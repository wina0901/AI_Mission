# LangChain 기반 연말정산 문서 질의응답 RAG 시스템

## 프로젝트 개요

이 프로젝트는 국세청 2024년 연말정산 신고 안내 PDF 문서를 기반으로 사용자의 질문에 답변할 수 있는 Retrieval-Augmented Generation(RAG) 시스템을 구축한 프로젝트입니다.

단순히 LLM에게 질문을 입력하는 방식이 아니라, 실제 연말정산 문서에서 질문과 관련된 내용을 먼저 검색하고, 검색된 문서 내용을 기반으로 답변을 생성하도록 구성했습니다.

이번 프로젝트에서는 LangChain을 활용하여 PDF 문서 로딩, 텍스트 전처리, 문서 청킹, 임베딩 생성, FAISS 벡터 데이터베이스 구축, Basic Vector RAG, BM25 기반 검색, Hybrid Search, MultiQuery Retrieval까지 단계적으로 구현했습니다.

또한 `Bllossom/llama-3.2-Korean-Bllossom-AICA-5B` 모델을 4bit 양자화 방식으로 로드하여 한국어 답변 생성을 수행했고, 여러 검색 방식에 따른 답변 품질 차이를 정성적으로 비교했습니다.

---

## 이 프로젝트를 왜 했는가

대규모 언어 모델은 자연어 생성 능력이 뛰어나지만, 특정 문서에 포함된 세부 정보나 최신 행정 문서 내용을 정확하게 답변하는 데에는 한계가 있습니다.

특히 연말정산 문서는 소득공제, 세액공제, 비거주자 공제 제한, 월세 세액공제, 기부금 공제 등 규정이 복잡하고 문서 분량도 많기 때문에, 사용자가 직접 필요한 내용을 찾기 어렵습니다.

이번 프로젝트는 이러한 문제를 해결하기 위해 진행했습니다.

- LangChain을 활용한 RAG 시스템의 전체 흐름을 직접 구현해보고 싶었습니다.
- 단순 LLM 답변이 아니라 실제 문서를 근거로 답변하는 구조를 만들고 싶었습니다.
- PDF 문서를 검색 가능한 형태로 전처리하고 벡터화하는 과정을 경험하고 싶었습니다.
- FAISS 기반 Dense Retrieval과 BM25 기반 Keyword Retrieval의 차이를 비교해보고 싶었습니다.
- Hybrid Retrieval과 MultiQuery Retrieval을 적용했을 때 검색 품질이 어떻게 달라지는지 확인하고 싶었습니다.
- 한국어 문서 기반 RAG에서 LLM 응답이 어떤 한계를 보이는지 직접 분석하고 싶었습니다.

즉, 이번 프로젝트는 단순한 챗봇 구현이 아니라, 외부 문서를 근거로 검색과 생성을 결합하는 실무형 LLM 응용 시스템을 직접 설계하고 검증하기 위해 진행했습니다.

## 문제 정의

- 주제: 국세청 연말정산 문서 기반 질의응답 시스템
- 문제 유형: Retrieval-Augmented Generation (RAG)
- 입력: 사용자의 한국어 자연어 질문
- 출력: 연말정산 문서를 기반으로 생성된 한국어 답변
- 사용 문서: `year_end_tax.pdf`
- 주요 구현 요소:
  - PDF Loading
  - Text Cleaning
  - Document Chunking
  - Embedding
  - Vector DB
  - Basic Vector RAG
  - BM25 Keyword Retrieval
  - Hybrid Retrieval
  - MultiQuery Retrieval

이번 프로젝트의 핵심은 LLM이 자체 지식만으로 답변하지 않도록 하고, 실제 문서에서 검색된 내용을 기반으로 답변을 생성하도록 만드는 것이었습니다.

## 프로젝트 진행 과정

### 1. 실행 환경 및 기본 설정

Colab 환경에서 프로젝트를 진행했으며, Google Drive에 저장된 PDF 파일을 불러와 사용했습니다.

주요 설정은 다음과 같습니다.

- 실행 환경: Google Colab
- 사용 디바이스: CUDA
- Torch 버전: 2.10.0+cu128
- 재현성을 위한 Seed: 42

필요한 패키지는 LangChain, Transformers, Sentence-Transformers, FAISS, PyPDF, BM25 등을 설치하여 사용했습니다.

---

### 2. PDF 문서 로드

`PyPDFLoader`를 사용해 PDF 문서를 페이지 단위 Document 객체로 불러왔습니다.

로드 결과는 다음과 같았습니다.

- 로드된 페이지 수: 426페이지
- 첫 페이지 내용: 연말정산 신고안내 표지 및 발간 정보
- metadata에는 source, total_pages, page 정보 포함

이 과정에서 PDF 문서가 정상적으로 페이지 단위로 분리되어 로드되는지 먼저 확인했습니다.

---

### 3. 문서 전처리

PDF에서 추출된 텍스트에는 줄바꿈, 탭, 불필요한 공백이 포함되어 있었기 때문에 전처리 함수를 정의해 문서를 정리했습니다.

적용한 전처리는 다음과 같습니다.

- 줄바꿈 제거
- 탭 제거
- 연속 공백 정리
- 빈 텍스트 페이지 제거
- metadata 유지

전처리 후 사용 가능한 페이지 수는 423페이지였습니다.

이 과정은 단순 문자열 정리가 아니라, 임베딩과 검색 품질을 높이기 위해 문서 내용을 일정한 형태로 정리하는 단계였습니다.

---

### 4. 청킹 실험

긴 PDF 문서를 그대로 검색 대상으로 사용하기 어렵기 때문에 `RecursiveCharacterTextSplitter`를 사용하여 문서를 청크 단위로 분할했습니다.

청킹 후보는 다음과 같이 설정했습니다.

| 설정 | Chunk Size | Chunk Overlap | 청크 수 |
|------|-----------:|--------------:|--------:|
| small | 500 | 100 | 1250 |
| medium | 800 | 150 | 813 |
| large | 1200 | 200 | 545 |

최종적으로는 `medium` 설정을 선택했습니다.

선택 이유는 다음과 같습니다.

- small은 청크 수가 많아 검색 단위가 지나치게 잘게 나뉠 수 있음
- large는 문맥은 많이 보존하지만 검색 단위가 커져 관련 없는 내용이 함께 포함될 수 있음
- medium은 문맥 보존과 검색 정밀도 사이의 균형이 가장 적절하다고 판단함

즉, 청킹은 단순히 문서를 자르는 작업이 아니라, 검색 정확도와 문맥 유지 사이의 균형을 조정하는 중요한 과정이었습니다.

---

### 5. 임베딩 모델 설정

한국어 문서의 의미 기반 검색을 위해 Hugging Face 임베딩 모델을 사용했습니다.

사용한 임베딩 모델은 다음과 같습니다.

```python
jhgan/ko-sroberta-multitask
```

임베딩 설정은 다음과 같습니다.

| 항목 | 값 |
|------|---:|
| 임베딩 모델 | jhgan/ko-sroberta-multitask |
| 임베딩 차원 | 768 |
| normalize_embeddings | True |
| 사용 디바이스 | CUDA |

이 모델을 사용한 이유는 한국어 문장 간 의미 유사도 계산에 적합하고, 연말정산 질문과 문서 청크 사이의 의미 기반 검색에 활용하기 좋다고 판단했기 때문입니다.

---

### 6. FAISS 벡터 데이터베이스 구축

전처리 및 청킹을 거친 813개의 문서 청크를 임베딩한 뒤 FAISS 벡터 데이터베이스에 저장했습니다.

구성은 다음과 같습니다.

- Vector Store: FAISS
- 저장 청크 수: 813개
- 검색 방식: similarity search
- 검색 개수: k=4

FAISS를 사용함으로써 대량의 문서 청크 중 질문과 의미적으로 가장 가까운 문서를 빠르게 검색할 수 있도록 구성했습니다.

### 7. 기본 검색 결과 확인

예시 질문으로 다음 질문을 사용했습니다.

```text
2024년 개정 세법 중 월세와 관련한 내용이 있을까?
```

기본 벡터 검색 결과, 월세액 세액공제 소득기준 및 한도 상향과 관련된 페이지가 상위 문서로 검색되었습니다.

검색된 주요 내용은 다음과 같습니다.

- 월세 세액공제 대상 소득기준 상향
- 총급여 7천만원 이하에서 8천만원 이하로 확대
- 종합소득금액 6천만원 이하에서 7천만원 이하로 확대
- 연간 월세액 공제한도 750만원에서 1,000만원으로 상향

이 결과를 통해 FAISS 기반 의미 검색이 질문과 관련된 문서를 비교적 정확하게 찾아낼 수 있음을 확인했습니다.

## LLM 설정

답변 생성을 위해 한국어 LLM을 사용했습니다.

사용 모델은 다음과 같습니다.

```python
Bllossom/llama-3.2-Korean-Bllossom-AICA-5B
```

모델 로드 및 생성 설정은 다음과 같습니다.

| 항목 | 설정 |
|------|------|
| 모델 | Bllossom/llama-3.2-Korean-Bllossom-AICA-5B |
| 로드 방식 | 4bit Quantization |
| quant type | nf4 |
| compute dtype | float16 |
| max_new_tokens | 450 |
| do_sample | False |
| repetition_penalty | 1.2 |
| no_repeat_ngram_size | 4 |

4bit 양자화를 적용한 이유는 Colab GPU 환경에서 5B 규모의 한국어 LLM을 보다 효율적으로 로드하기 위해서였습니다.

또한 반복 답변을 줄이기 위해 `repetition_penalty`와 `no_repeat_ngram_size`를 설정했습니다.

## 프롬프트 설계

RAG 답변의 신뢰성을 높이기 위해 프롬프트에 다음 조건을 포함했습니다.

- 국세청 2024년 연말정산 신고 안내 문서를 기반으로 답변
- 제공된 맥락만 사용
- 문서에 없는 내용은 확인할 수 없다고 답변
- 반드시 한국어로 작성
- 3줄 이내로 답변
- 예시, 원문 복사, 반복 설명 금지
- 답변 마지막 줄에 참고 페이지 작성

이 프롬프트는 LLM이 검색된 문서 외의 내용을 임의로 생성하지 않도록 제한하기 위한 목적이었습니다.

## Basic Vector RAG 구현

기본 RAG 체인은 다음과 같은 흐름으로 구성했습니다.

사용자 질문

→ FAISS Vector Retriever

→ 관련 문서 4개 검색

→ 검색 문서 포맷팅

→ Prompt Template 입력

→ LLM 답변 생성

→ 출력 후처리

추가로 LLM 출력에서 깨진 문자열이나 반복 문장을 줄이기 위해 `clean_answer` 함수를 구현했습니다.

Basic Vector RAG는 가장 단순한 구조이지만, 의미 기반 검색을 통해 질문과 관련된 문서를 찾고 답변을 생성할 수 있었습니다.

---

## Hybrid Search 구현

기본 벡터 검색은 의미적으로 유사한 문서를 찾는 데 유리하지만, 특정 키워드가 중요한 문서 검색에서는 한계가 있을 수 있습니다.

이를 보완하기 위해 다음 두 검색기를 결합했습니다.

- FAISS Vector Retriever
- BM25 Keyword Retriever

`EnsembleRetriever`를 사용하여 두 검색기를 결합했으며, 가중치는 다음과 같이 설정했습니다.

| 검색 방식 | 가중치 |
|----------|------:|
| Vector Retriever | 0.6 |
| BM25 Retriever | 0.4 |

Hybrid Search를 통해 의미 기반 검색과 키워드 기반 검색의 장점을 함께 활용하고자 했습니다.

예를 들어 월세 세액공제 질문에서는 월세액 세액공제 소득기준 및 한도 상향 페이지가 상위에 검색되었고, 관련 세액공제 문서도 함께 검색되었습니다.

---

## MultiQuery Retrieval 구현

MultiQuery Retrieval은 사용자의 질문 하나를 그대로 검색하는 대신, LLM을 활용해 질문을 여러 형태로 변환한 뒤 검색하는 방식입니다.

이번 프로젝트에서는 `MultiQueryRetriever.from_llm`을 사용해 MultiQuery 검색기를 구성했습니다.

예시 질문은 다음과 같습니다.

```text
비거주자가 연말정산에서 주의해야 할 공제 제한은?
```

검색 결과 다음과 같은 문서가 상위에 검색되었습니다.

- 외국인 거주자와 비거주자의 연말정산 소득·세액공제 비교
- 비거주자의 국내원천소득 및 연말정산 방법

이를 통해 MultiQuery Retrieval이 단일 질문 검색보다 더 다양한 관련 문맥을 검색할 수 있음을 확인했습니다.

다만 생성 단계에서는 답변이 길어질수록 반복 표현이나 불안정한 문장이 나타나는 한계도 함께 확인했습니다.

## 정성 평가

다음 질문들을 사용해 Basic Vector RAG, Hybrid RAG, MultiQuery RAG의 답변을 비교했습니다.

- 연말 정산 때 비거주자가 주의할 점을 알려 줘.
- 2024년 개정 세법 중에 월세와 관련한 내용이 있을까?
- 의료비 세액공제를 받을 때 주의해야 할 점을 알려 줘.

정성 평가 결과는 다음과 같이 정리할 수 있습니다.

### Basic Vector RAG

Basic Vector RAG는 구조가 단순하지만 비교적 안정적으로 관련 문서를 검색했습니다.

비거주자 질문에서는 비거주자의 국내원천소득과 연말정산 방법에 관한 문서를 검색했으며, 월세 질문에서는 월세액 세액공제 개정 내용을 포함한 문서를 검색했습니다.

다만 LLM 생성 과정에서 일부 단어가 깨지거나 문장이 어색하게 생성되는 문제가 있었습니다.

### Hybrid RAG

Hybrid RAG는 FAISS와 BM25를 결합하여 검색 결과의 다양성을 확보할 수 있었습니다.

특히 월세 세액공제처럼 특정 키워드가 중요한 질문에서는 관련 문서를 비교적 잘 검색했습니다.

Dense Retrieval과 Keyword Retrieval을 함께 사용했기 때문에 검색 정확도와 정보 다양성 측면에서 가장 균형적인 방식으로 판단했습니다.

### MultiQuery RAG

MultiQuery RAG는 질문을 여러 관점으로 확장하여 더 다양한 문서를 검색할 수 있었습니다.

비거주자 공제 제한 질문에서는 비거주자와 외국인 거주자의 공제 차이를 설명하는 부록 문서와 비거주자 연말정산 방법 문서를 함께 검색했습니다.

하지만 생성 답변에서는 반복 표현, 부정확한 표현, 문맥 이탈이 발생하는 경우가 있어 생성 안정성 측면에서는 한계가 있었습니다.

---

## 최종 결과 해석

이번 프로젝트에서는 Basic Vector Retrieval, BM25 Keyword Retrieval, Hybrid Retrieval, MultiQuery Retrieval을 단계적으로 구현하고 비교했습니다.

최종적으로 확인한 내용은 다음과 같습니다.

- PDF 문서를 전처리한 뒤 423페이지의 텍스트 문서로 정리할 수 있었습니다.
- RecursiveCharacterTextSplitter를 활용해 3가지 청킹 설정을 비교했고, 최종적으로 medium 설정을 선택했습니다.
- `jhgan/ko-sroberta-multitask` 임베딩 모델을 사용해 768차원 문서 임베딩을 생성했습니다.
- FAISS에 813개의 청크를 저장하고 의미 기반 검색을 수행했습니다.
- BM25를 추가해 키워드 기반 검색을 구성했습니다.
- Hybrid Retrieval은 검색 정확도와 정보 다양성 측면에서 가장 균형적인 결과를 보였습니다.
- MultiQuery Retrieval은 관련 문서를 더 넓게 찾는 데 도움이 되었지만, 생성 답변 안정성은 상대적으로 낮았습니다.
- Bllossom Korean LLM은 한국어 답변 생성이 가능했지만, 일부 긴 답변에서는 반복, 깨진 단어, 문맥 이탈 문제가 발생했습니다.

즉, 이번 프로젝트를 통해 RAG 시스템에서는 LLM 자체의 성능뿐 아니라 문서 전처리, 청킹, 검색기 구성, 프롬프트 설계, 출력 후처리가 모두 중요하다는 점을 확인했습니다.

## 이 프로젝트를 통해 갖추게 된 능력

### 1. RAG 파이프라인 구현 능력

PDF 문서 로드부터 전처리, 청킹, 임베딩, 검색, 생성까지 RAG 시스템의 전체 흐름을 직접 구현할 수 있게 되었습니다.

### 2. LangChain 활용 능력

LangChain의 Document Loader, Text Splitter, Retriever, PromptTemplate, Runnable, OutputParser를 활용하여 실제 동작하는 RAG 체인을 구성했습니다.

### 3. PDF 문서 전처리 능력

PDF에서 추출된 텍스트를 정리하고 빈 페이지를 제거하며, 검색에 적합한 문서 형태로 변환하는 과정을 경험했습니다.

### 4. 청킹 전략 설계 능력

chunk size와 overlap을 다르게 설정해 청크 수와 문맥 보존 정도를 비교했습니다.

이를 통해 RAG에서 청킹 전략이 검색 품질과 답변 품질에 직접적인 영향을 줄 수 있다는 점을 이해했습니다.

### 5. 임베딩 기반 검색 시스템 구축 능력

한국어 임베딩 모델을 활용해 문서 청크를 벡터화하고, FAISS를 이용해 의미 기반 검색 시스템을 구축했습니다.

### 6. 검색 방식 비교 능력

FAISS 기반 Dense Retrieval, BM25 기반 Keyword Retrieval, Hybrid Retrieval, MultiQuery Retrieval을 비교하면서 검색 방식별 장단점을 분석했습니다.

### 7. 한국어 LLM 활용 능력

Bllossom Korean LLM을 4bit 양자화로 로드하고, LangChain Pipeline과 연결하여 한국어 답변 생성을 수행했습니다.

### 8. RAG 응답 품질 분석 능력

검색 결과와 생성 답변을 함께 확인하면서, 검색은 잘 되었지만 생성이 불안정한 경우와 검색 자체가 부족한 경우를 구분해 해석할 수 있게 되었습니다.

## 이 프로젝트에서 중요하게 본 점

이번 프로젝트에서 가장 중요하게 생각한 점은 단순히 LLM이 답변을 생성하게 하는 것이 아니라, 실제 문서 검색 결과를 기반으로 답변하도록 만드는 것이었습니다.

그래서 다음과 같은 원칙으로 프로젝트를 진행했습니다.

- PDF 문서를 먼저 정확하게 로드하고 전처리하기
- 청킹 설정을 여러 방식으로 비교하기
- 한국어 문서에 적합한 임베딩 모델 사용하기
- FAISS를 활용해 의미 기반 검색 구조 만들기
- BM25를 추가해 키워드 기반 검색 보완하기
- Hybrid Retrieval로 검색 안정성 높이기
- MultiQuery Retrieval로 검색 범위 확장하기
- 프롬프트와 후처리를 통해 답변 형식을 제어하기
- 검색 결과와 생성 결과를 분리해서 평가하기

이러한 흐름은 향후 사내 문서 검색, 정책 문서 QA, 매뉴얼 챗봇, 법률·세무 문서 질의응답 같은 RAG 기반 서비스 개발에 중요한 기반이 될 수 있다고 생각합니다.

## 한계점 및 개선 방향

이번 프로젝트를 통해 RAG 시스템의 기본 구조는 구현할 수 있었지만, 다음과 같은 한계도 확인했습니다.

### 1. 생성 답변의 안정성 문제

일부 질문에서 LLM이 깨진 단어를 생성하거나, 문맥과 맞지 않는 표현을 출력하는 문제가 있었습니다.

이를 개선하기 위해서는 더 강력한 한국어 LLM을 사용하거나, generation parameter를 추가로 조정할 필요가 있습니다.

### 2. 근거 페이지 출력의 불안정성

프롬프트에는 참고 페이지를 출력하도록 했지만, 실제 생성 결과에서는 페이지 정보가 정확하게 정리되지 않는 경우가 있었습니다.

향후에는 LLM이 직접 페이지를 생성하게 하기보다, 검색된 metadata를 별도 로직으로 추출해 답변과 함께 표시하는 방식이 더 안정적일 수 있습니다.

### 3. 정량 평가 부족

이번 프로젝트에서는 정성 평가 중심으로 RAG 방식을 비교했습니다.

향후에는 질문-정답 데이터셋을 직접 구축하고, Retrieval Precision, Recall, Answer Correctness, Faithfulness 같은 지표로 정량 평가를 추가할 수 있습니다.

### 4. Re-ranking 미적용

FAISS와 BM25 검색 결과를 바로 LLM에 전달했기 때문에, 검색된 문서 중 실제로 가장 적합한 문서를 다시 정렬하는 과정은 포함하지 않았습니다.

향후 Cross Encoder 기반 Re-ranker를 추가하면 검색 품질을 더 높일 수 있습니다.

### 5. 문서 구조 활용 부족

PDF의 목차, 장 제목, 표 구조를 충분히 활용하지 못했습니다.

향후에는 페이지별 제목, 섹션 정보, 표 데이터를 metadata로 추가해 검색 품질을 개선할 수 있습니다.
